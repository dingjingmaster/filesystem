## 1. 问题与边界

- 问题描述：`filesystem-gui/src/main.rs` 已增长到 3600 行以上，同时承载窗口配置、应用状态机、显示模型、后台任务、主题图标解析、UI 组件、工具函数、样式和测试，继续堆叠会提高后续文件操作、属性页和视图扩展的维护成本。
- 调研目的：在不改变当前功能、不引入新依赖的前提下，拆出清晰模块边界，使状态、任务、视图组件和工具函数各自可维护。
- 包含：GUI crate 内部模块拆分、可见性调整、模块职责说明、现有测试迁移。
- 不包含：修改 `filesystem-core` API、改变 iced/wgpu 依赖、改变窗口/文件操作/属性弹窗行为、补充新功能。
- 非目标：不做 UI 重设计，不重写状态机，不引入插件框架或复杂分层。
- 禁止触碰范围：不回退用户已有功能，不执行破坏性文件操作，不修改 `AGENTS.project.md`，不做无关格式化。

## 2. 当前证据

- 现有实现/现状：
  - `crates/filesystem-gui/src/main.rs` 约 3620 行。
  - 顶部常量和窗口 icon 配置与业务状态混在同一文件。
  - `FileManager` 同时负责 update/view 编排、导航历史、搜索、选择、重命名、属性弹窗和权限编辑。
  - 后台任务函数、主题图标解析、UI helper、纯格式化/几何工具和 `style` 模块均内联在 `main.rs`。
  - GUI 已有单元测试覆盖流式图标列数、菜单钳制、右键菜单语义、属性弹窗位置、权限切换和重命名长度限制。
- 已知约束：
  - GUI 继续使用 iced 的消息驱动架构。
  - 所有可能阻塞 UI 的文件系统任务继续通过 `Task::perform`。
  - 不新增依赖，保持 wgpu/X11/Wayland 构建边界。
  - 模块拆分必须能通过 `cargo fmt --check`、`cargo check -p filesystem-gui`、`cargo test -p filesystem-core`、`cargo test -p filesystem-gui` 和 `make test`。
- 关键日志/数据/用户反馈：用户明确要求“按模块划分拆分代码，确保功能不受影响，拆分模块要考虑使用设计模式，要做到可持续维护”。
- Bug 证据等级：不适用，本次为架构重构。
- 证据不足项：无真实 GUI 会话自动化截图验证；本次不改变 UI 行为，主要依赖编译、测试和人工 diff 审查。

## 4. 候选方案

| 方案 | 核心思路 | 优点 | 风险/代价 | 适用条件 |
|------|----------|------|-----------|----------|
| A | 仅把 `style` 拆出 | 改动最小 | 无法解决状态、任务、模型和组件混杂，维护收益不足 | 只需要快速降低文件长度 |
| B | 按职责拆成 `app`、`model`、`config`、`tasks`、`icons`、`components`、`utils`、`style` | 边界清晰，不改 iced 消息架构，不新增依赖，适合持续扩展 | 需要较多可见性调整，必须严控行为不变 | 当前需求 |
| C | 引入更深的 MVC/MVVM 或 command bus 抽象 | 长期可扩展性强 | 过度设计，容易改变行为并增加维护成本 | 后续功能复杂到单一状态机不可维护时 |

## 5. 推荐结论

- 推荐方案：采用方案 B，做职责型模块拆分，保留 iced 当前 Elm/Redux 风格消息循环。
- 取舍理由：
  - `app` 保留 `FileManager` 状态机和 view 编排，避免在一次重构中改业务控制流。
  - `model` 放共享状态和消息类型，减少模块间循环依赖。
  - `tasks` 作为后台任务适配层，延续“阻塞 I/O 不进 UI 线程”的约束。
  - `icons` 使用 Resolver/Strategy 风格封装主题图标查找和回退图标。
  - `components` 放可复用 iced widget 工厂，便于后续属性页、菜单和视图控件维护。
  - `utils` 放纯函数和测试，降低 UI 代码噪音。
  - `style` 独立承载主题样式，避免组件与颜色/边框策略混杂。
- 需要进入 Plan 的关键约束：
  - 不改公开功能和 core API。
  - 不新增依赖。
  - `main.rs` 只保留入口和模块声明。
  - 拆分后测试从 monolithic `main.rs` 迁移到对应模块。
- 需要用户确认的问题：无。
- 后续验证方向：完整格式检查、GUI crate 编译、core/gui 测试、Makefile 测试、diff whitespace 检查。

## 6. L4 审视

- 安全审查：不涉及破坏性操作，不触碰用户数据；风险集中在模块可见性和异步任务消息类型签名，需编译和测试验证。
- 产品审查：用户目标是维护性重构，不应改变任何交互或视觉表现。
- 架构审查：职责型拆分符合当前 iced 架构，避免引入新的框架层。
- 工程审查：应优先机械迁移和最小可见性调整，不做趁机重写。
