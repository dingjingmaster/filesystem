# 文件视图右键菜单、写操作与属性弹窗调研报告

> 文档元数据
> - 文件编号：2
> - 文档类型：research
> - 文件路径：docs/dev/2-research-context-menu-file-ops-properties.md
> - 文档版本：v1.0.0
> - 最后更新：2026-06-25
> - 关联需求：图标视图和列表视图空白处右键菜单；支持新建文件/文件夹、粘贴、全选、在终端打开、查看当前文件夹属性。

## 1. 问题与边界

- 问题描述：当前 GUI 只能只读浏览、搜索、选择和切换视图；缺少文件视图空白区右键菜单、首批写操作、外部终端打开和当前文件夹属性查看。
- 调研目的：确认在不依赖系统桌面服务的前提下，基于现有 iced GUI 和 core 模块如何实现这些功能，并明确剪贴板/权限/属性展示的首版边界。
- 包含：
  - 文件视图空白处右键弹出覆盖菜单，不改变文件布局。
  - 新建空文件、新建目录，并进入内联重命名状态。
  - 从剪贴板粘贴本地路径或 `file://` URI 对应的文件/文件夹。
  - 全选当前视图列出的条目。
  - 按 `terminator`、`mate-terminal`、`gnome-terminal` 顺序从 `PATH` 查找并在当前目录启动。
  - 弹窗查看当前文件夹属性、权限概览和内容权限参考页。
- 不包含：
  - 删除、永久删除、回收站。
  - 完整系统文件管理器剪贴板 MIME target 支持。
  - 权限修改、递归 chmod/chown。
  - 通过 DBus/GVFS/portal/xdg-open/MIME 服务打开终端或文件。
- 非目标：不实现选中文件/文件夹的右键菜单；本轮只处理文件视图空白处菜单。
- 禁止触碰范围：不修改 `/data/source/cosmic-files`；不对真实用户目录执行测试性复制/移动/删除；不调用 shell 拼接命令。

## 2. 当前证据

- 现有实现/现状：
  - `filesystem-core` 已提供 `scan_dir`、`search_file_names` 和临时目录测试夹具。
  - `filesystem-gui` 使用 iced 0.14；目录扫描、搜索和图标解析已通过 `Task::perform` 后台执行。
  - 文件视图主区域已有 `mouse_area`，当前处理左键框选和滚动。
  - 主内容已有 `stack` 覆盖层模式，可复用为右键菜单和属性弹窗，避免菜单影响文件布局。
- 已知约束：
  - `iced_widget::MouseArea` 支持 `on_right_press` 和 `on_right_release`，可以捕获右键。
  - `iced::clipboard::read()` 只能读取标准剪贴板里的文本字符串，不能选择 `text/uri-list`、`x-special/gnome-copied-files` 等 MIME target。
  - 因此首版粘贴只能解析 iced 能读到的文本：绝对路径、`file://` URI、`copy\nfile://...`、`cut\nfile://...`、`text/uri-list` 风格多行内容。
  - 终端启动可用 `std::process::Command` 直接传 argv；三个目标终端都支持 `--working-directory=PATH`。
  - 文件夹属性递归统计可能很慢，必须放后台任务。
- 关键日志/数据/用户反馈：用户要求任何可能阻塞 UI 的任务放到线程里执行；本需求包含文件复制、移动、创建、属性统计和终端探测。
- Bug 证据等级：不适用，新功能需求。
- 证据不足项：
  - 真实 GNOME/KDE/COSMIC 文件管理器复制/剪切后的剪贴板，iced 是否能读到文本 fallback 取决于运行时剪贴板实现和源应用提供的 target。
  - 未在真实 X11/Wayland 会话验证右键弹窗定位和剪贴板读取。

## 3. 安全门禁摘要

| 项 | 结论 |
|----|------|
| 风险矩阵初判 | L4，新系统继续引入文件系统写操作、外部进程和递归目录统计 |
| 命令权限 | C1，本轮只修改源码/文档并在临时目录自动化测试；不执行真实用户目录破坏性操作 |
| 高风险开发门禁 | 是，文件系统写操作、移动、递归复制、外部进程、错误路径 |
| 破坏性操作 | 产品功能含移动语义；验证必须限制在测试临时目录 |
| 用户已有修改 | 是，上一轮 GUI/文档变更仍在工作区；继续在其基础上修改，不回退 |
| 用户确认事项 | 无需阻塞；首版剪贴板支持范围需在计划和最终结果中说明 |

## 4. 候选方案

| 方案 | 核心思路 | 优点 | 风险/代价 | 适用条件 |
|------|----------|------|-----------|----------|
| A | 仅使用 iced 文本剪贴板，解析本地路径/URI；core 实现文件创建、重命名、复制/移动和属性统计 | 无新增剪贴板依赖；符合少依赖目标；易测试 | 无法保证读取所有桌面文件管理器的专用 MIME 剪贴板格式 | 首版可接受文本 fallback；保持无桌面服务依赖 |
| B | 引入专门剪贴板库或按 X11/Wayland 协议读取 MIME target | 更接近桌面文件管理器粘贴行为 | 新依赖和平台复杂度高；Wayland/X11 差异大；不一定保持少依赖 | 后续明确要求完整外部文件剪贴板兼容时 |
| C | 通过 `wl-paste`、`xclip`、`xclip-copyfile` 等外部二进制读取剪贴板 | 可覆盖更多环境；不直接引入 Rust 依赖 | 引入运行时二进制依赖和协议差异；需要探测降级 | 用户接受剪贴板工具作为可选依赖时 |

## 5. 推荐结论

- 推荐方案：首版采用方案 A。
- 取舍理由：
  - 继续遵守“越少依赖越好”和“不依赖系统桌面服务”。
  - core 写操作可用临时目录单元测试覆盖，风险可控。
  - 粘贴能力先覆盖最常见文本路径/URI fallback，不把剪贴板协议复杂度引入本轮。
- 需要进入 Plan 的关键约束：
  - 所有写操作、递归统计、粘贴和终端探测都通过 `Task::perform` 后台任务执行。
  - 复制/移动不覆盖已有文件，目标存在时自动生成唯一文件名。
  - 移动首版使用 `fs::rename`，跨设备移动不做复制后删除退化，避免未经专项测试的高风险删除。
  - 属性弹窗首版只查看，不修改权限。
- 需要用户确认的问题：无阻塞项；最终交付必须说明剪贴板首版边界。
- 后续验证方向：core 临时目录测试覆盖创建、重命名、防覆盖、递归复制、移动、剪贴板文本解析、属性统计；GUI 执行 `cargo check` 和 workspace 测试。
