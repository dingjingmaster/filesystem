# 大目录打开性能优化调研报告

> - 关联需求：解释并优化“文件夹中有大量文件时候，打开会卡一瞬间”的问题。

## 1. 问题与边界

- 问题描述：大目录打开时会卡一瞬间，主要风险来自等待 `read_dir` 扫描结束、逐个文件 MIME/主题图标解析，以及一次性构建全部图标/列表 widget。
- 调研目的：让大目录尽快显示可见内容，并把耗时操作分批、后台化、缓存化。
- 包含：目录分批加载、基础条目先显示、MIME/图标后台装饰、主题图标缓存、图标视图和列表视图虚拟化。
- 不包含：真实异步文件系统 watcher、分页持久缓存、磁盘索引服务、数据库、桌面服务集成。
- 非目标：不改变右键菜单、双击打开、搜索语义和文件操作行为。
- 禁止触碰范围：不引入 DBus/GVFS/portal/XDG 服务依赖；不改系统配置；不执行破坏性文件操作验证。

## 2. 当前证据

- 现有实现/现状：目录加载在后台任务中执行，但完成消息在 `scan_dir` 和 `decorate_entries` 全部结束后才到 UI；`decorate_entries` 会对文件执行 `detect_path` 和主题图标读取；`grid`/`list` 会为全部条目创建 widget。
- 已知约束：所有可能阻塞 UI 的文件系统任务必须在后台执行；项目目标是少依赖、Linux 本地 GUI 文件管理器。
- 关键日志/数据/用户反馈：用户反馈“大量文件时候，打开会卡一瞬间”，并认可按“先显示、缓存、虚拟化、分批加载”优化。
- Bug 证据等级（按需）：E1，有用户现象反馈和代码路径证据，但未提供固定数量目录的性能基准。
- 证据不足项：未做真实 X11/Wayland 大目录帧时间测量。

## 4. 候选方案

| 方案 | 核心思路 | 优点 | 风险/代价 | 适用条件 |
|------|----------|------|-----------|----------|
| A | 保持完成后一次性返回，只优化图标缓存 | 改动小 | 首屏仍等待扫描和全部装饰，无法解决大目录打开卡顿 | 目录规模较小 |
| B | 目录扫描分批返回基础条目，MIME/图标后台装饰，视图虚拟化 | 首屏更快，UI 节点数量稳定，不新增依赖 | 状态机更复杂，需要丢弃过期装饰结果 | 当前需求 |
| C | 引入索引/缓存数据库 | 后续重复访问更快 | 新依赖和持久化复杂度高，不符合当前少依赖目标 | 未来大量目录索引需求 |

## 5. 推荐结论

- 推荐方案：采用方案 B。
- 取舍理由：它直接对应用户提出的四点优化，不引入新依赖，不改变产品语义；分批和虚拟化分别解决“等待数据”和“一次性渲染过多节点”两个瓶颈。
- 需要进入 Plan 的关键约束：后台任务必须带请求 ID；过期批次和装饰结果必须丢弃；基础图标不能读取文件内容；主题图标缓存应为进程内只读缓存。
- 需要用户确认的问题：无。
- 后续验证方向：格式化、core/mime/gui 测试、GUI 编译、`make test`、人工大目录打开体验验证。
