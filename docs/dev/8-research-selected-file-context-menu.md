# 选中文件右键菜单调研报告

> 文档元数据
> - 文件编号：8
> - 文档类型：research
> - 文件路径：docs/dev/8-research-selected-file-context-menu.md
> - 文档版本：v1.0.0
> - 最后更新：2026-06-25
> - 关联需求：左键选中文件后，右键显示文件菜单，支持默认应用打开、打开方式、复制、剪切、重命名、删除和属性

## 1. 问题与边界

- 问题描述：当前只支持空白菜单和单个文件夹菜单；文件右键菜单、默认应用打开、打开方式列表和文件属性缺失。
- 调研目的：在不依赖桌面服务的前提下，补齐单个选中文件右键菜单及相关文件操作。
- 包含：
  - 单个已选中文件右键菜单。
  - 默认应用名称显示和直接打开。
  - 打开方式弹窗，列出本地 `.desktop` 中声明可打开该 MIME 类型的应用，默认应用预选。
  - 文件复制、剪切、重命名、删除。
  - 文件属性弹窗，参考用户给出的文件属性截图。
- 不包含：
  - 写入默认应用设置。
  - 文件多选批量菜单。
  - 通过 DBus/GVFS/portal/XDG MIME 服务查询或打开。
  - 回收站。
- 非目标：不新增外部依赖，不实现完整 MIME 数据库。
- 禁止触碰范围：不执行真实用户目录删除；不依赖桌面服务；不修改 Cargo 依赖。

## 2. 当前证据

- 现有实现/现状：
  - 文件夹菜单已实现 `ContextMenuState::Folder`、内部 `ClipboardState`、删除确认和后台删除任务。
  - core 已有 `paste_paths`、`rename_entry`、`delete_entry`，可用于文件复制/剪切、重命名和删除。
  - core 只有 `FolderProperties`，没有文件属性结构。
  - GUI 文件双击当前没有配置打开器。
  - 现有设计要求不依赖 DBus/GVFS/portal/XDG MIME 服务。
- 已知约束：
  - 所有文件操作和应用启动都应走后台任务。
  - 默认应用和可选应用只能通过本地文件解析，首版不持久化默认应用。
  - 删除前必须弹窗确认。
- 关键日志/数据/用户反馈：用户给出文件菜单项和两张参考图，文件属性图展示摘要信息，打开方式图展示应用列表和默认选中。
- Bug 证据等级：不适用，本次为功能实现。
- 证据不足项：不同发行版 `.desktop` 和 `mimeapps.list` 分布差异较大，首版按 XDG 常见路径解析。

## 3. 安全门禁摘要

| 项 | 结论 |
|----|------|
| 风险矩阵初判 | L4，涉及文件系统写操作、外部程序启动和 GUI 交互扩展 |
| 命令权限 | C0/C1；不需要 C2/C3 |
| 高风险开发门禁 | 是，文件系统写操作和外部程序启动 |
| 破坏性操作 | 代码能力包含删除；自动化验证只在临时目录内执行 |
| 用户已有修改 | 编码前工作区干净 |
| 用户确认事项 | 无；运行时删除仍由确认弹窗二次确认 |

## 4. 候选方案

| 方案 | 核心思路 | 优点 | 风险/代价 | 适用条件 |
|------|----------|------|-----------|----------|
| A | 调用 `xdg-open` 和 `xdg-mime` | 实现快，系统行为一致 | 依赖外部桌面工具/服务语义，不符合项目边界 | 允许桌面服务时 |
| B | 解析本地 `mimeapps.list` 和 `.desktop`，直接启动 Exec | 不依赖桌面服务，符合独立边界 | MIME 覆盖不完整，Exec 解析需保守实现 | 当前需求 |
| C | 内置完整 MIME 和 desktop-entry 解析库 | 行为更完整 | 新依赖和复杂度较高 | 后续需要高兼容时 |

## 5. 推荐结论

- 推荐方案：采用方案 B。
- 取舍理由：
  - 只读取本地配置和应用桌面文件，符合“不依赖系统桌面服务”的长期边界。
  - 应用注册表可启动时后台加载，避免右键菜单阻塞 UI。
  - 打开方式弹窗可以复用已加载的应用注册表。
- 需要进入 Plan 的关键约束：
  - 首版 MIME 通过扩展名映射覆盖常见类型；未知类型使用 `application/octet-stream`。
  - 默认应用优先读取 `mimeapps.list` 的 `[Default Applications]`。
  - 应用列表来自 `.desktop` 的 `MimeType`，只展示与该 MIME 匹配的应用。
  - `Exec` 字段只做保守 field-code 替换和直接 `Command::spawn`。
  - 不写默认应用设置。
- 需要用户确认的问题：无。
- 后续验证方向：core 文件属性测试、GUI 编译测试、应用解析纯函数测试、完整 Makefile 测试；真实 GUI 和真实应用打开需人工验证。
