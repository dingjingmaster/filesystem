## 1. 问题定义

- 问题现象：属性权限页所有者已显示为 `用户名(实际UID)`，但用户组没有正确显示为 `组名字(实际GID)`。
- 预期行为：用户组行优先显示当前 GID 对应的系统组名，并保留实际 GID，例如 `dingjing(999)`。
- 影响范围：文件夹右键属性 -> 权限页中的用户组值显示。
- 不包含：不修改文件 owner/group，不修改权限保存逻辑，不改变 core 属性数据结构。

## 2. 证据与根因

- 复现方式：用户反馈属性权限页“所有者”已正确，“用户组”选项未正确显示。
- 证据等级：E1。
- 关键日志/堆栈/输入：本地代码检查确认用户组只通过 `/etc/group` 解析 GID，解析失败时会回退 `GID xxx`。
- 根因：用户组名称解析只依赖 `/etc/group` 文件缓存，未优先使用系统 NSS 的 GID 解析路径；在组信息不完全来自 `/etc/group` 的环境中可能无法得到组名。
- 相关代码路径：`crates/filesystem-gui/src/utils.rs` 的 `group_label()`。

## 3. 修复方案

- 最小修复点：只修改 GUI 层用户组显示函数。
- 代码逻辑改动：`group_label()` 先通过 `libc::getgrgid_r` 按 GID 查询系统组名；失败后再读取 `/etc/group` 兜底；仍失败时保留 `GID xxx`。
- 影响的使用场景：属性权限页用户组值显示。
- 不影响的使用场景：所有者显示、列表视图所有者列、权限 mode 修改、core 属性读取。

## 7. 修复总结

- 最终结果：用户组显示改为优先通过系统 GID 解析组名，成功时显示 `组名字(实际GID)`。
- 计划偏差：原 L1 显示调整只覆盖 `/etc/group` 文件解析；本次按用户反馈补强为 L2 FFI 查询。
