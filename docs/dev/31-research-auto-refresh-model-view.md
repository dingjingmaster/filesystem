# 自动刷新 model/view 优化调研报告

> 文档元数据
> - 文件编号：31
> - 文档类型：research
> - 文件路径：docs/dev/31-research-auto-refresh-model-view.md
> - 文档版本：v1.0.0
> - 最后更新：2026-07-14
> - 关联需求：编译导致频繁刷新文件夹、界面闪烁；按目录数据刷新和可视范围刷新方向优化。

## 1. 问题与边界

- 问题描述：当前文件夹自动刷新把所有相关 `notify` 事件合并成 `CurrentFolderChanged(cwd)`，200ms 后复用 `reload()`；`reload()` 会清空 `entries`、重置滚动/选择和分批重建 UI，编译等高频文件事件会造成文件夹闪烁。
- 调研目的：将自动刷新从全量清空 reload 调整为接近 model/view 的数据更新；切换目录仍保持完整刷新。
- 包含：当前目录直接子项事件分类、单项条目刷新、结构变化后台快照同步、可视范围影响判断、状态机测试。
- 不包含：递归监听子目录内部变化、硬编码忽略 `target`/`build` 等目录、改变切换目录加载语义、引入新依赖。
- 非目标：不实现完整 Qt model/view 框架；不做跨平台文件事件精准语义抽象。
- 禁止触碰范围：不修改 `AGENTS.project.md`；不做无关 UI 重构；不改变文件写操作语义。

## 2. 当前证据

- 现有实现/现状：
  - `watch_current_folder` 使用 `notify` 非递归监听当前 `cwd`，事件命中后发送 `CurrentFolderChanged(cwd)`。
  - `should_refresh_for_event` 对 `Create`、`Remove`、大多数 `Modify`、`Close(Write)` 和 `need_rescan()` 返回 true。
  - `CurrentFolderRefreshReady` 触发 `reload()`；`reload()` 的 `clear_for_new_directory()` 会立即清空列表并重置浏览状态。
  - 图标视图和列表视图已有可视范围虚拟化，但数据层刷新粒度仍是整个目录。
- 已知约束：
  - 当前排序按目录优先和文件名排序，不按大小或修改时间排序；普通文件内容写入通常只需更新单项 metadata。
  - 用户要求数据层只关注当前目录变化，不包含当前目录下子目录内部修改。
  - 编译期间事件可能高频、重复、乱序或不完整，必须保留不确定事件的目录级校准。
- 关键日志/数据/用户反馈：用户反馈编译导致频繁刷新文件夹、界面闪烁，并明确尾部去抖不合适。
- Bug 证据等级：E1。已有现象描述和代码路径证据，未在真实 GUI 会话中录制复现。
- 证据不足项：不同文件系统和 notify 后端对 rename、目录 metadata、队列溢出的具体事件形态未逐一验证。

## 4. 候选方案

| 方案 | 核心思路 | 优点 | 风险/代价 | 适用条件 |
|------|----------|------|-----------|----------|
| A | 尾部去抖，事件安静后统一 reload | 代码少 | 长编译期间目录状态滞后；用户已否定 | 不采用 |
| B | 自动刷新后台扫描，扫描完成后整体替换列表 | 不清空界面，能校准不完整事件 | 仍会扫描整个目录；不可见变化也会触发状态更新 | 可作为结构变化 fallback |
| C | 事件级单项更新 + 结构变化快照同步 | 普通写入只刷新单项；新增/删除/重命名再校准目录 | 事件分类和错误路径更复杂 | 推荐 |
| D | 完整增量 model/view 行插入删除移动 | 体验最佳 | 当前 iced 状态机改动大，超出本次必要范围 | 后续演进 |

## 5. 推荐结论

- 推荐方案：采用 C，并保留 B 作为不确定/结构事件的目录级后台快照同步；切换目录继续走现有全量加载。
- 取舍理由：普通编译写文件只更新对应条目的 metadata，不触发清空；新增、删除、重命名或不确定事件仍能通过快照校准保证正确性；不硬编码忽略构建目录。
- 需要进入 Plan 的关键约束：
  - watcher 需要传递事件 kind 和 paths，而不是只传 `cwd`。
  - 只处理当前目录直接子项；子目录内部路径忽略。
  - 目录子项的 modify/access 不触发父目录刷新；目录 create/remove/rename 仍是结构变化。
  - 自动刷新不调用会清空界面的 `reload()`；切换目录继续调用现有路径。
  - 可视范围外的单项 metadata 更新只更新数据模型，不重置界面状态。
- 需要用户确认的问题：无。
- 后续验证方向：`cargo test -p filesystem-core`、`cargo test -p filesystem-gui`、`cargo check -p filesystem-gui`、`cargo fmt --check`。
