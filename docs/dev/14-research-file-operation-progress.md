# 文件复制剪切进度调研报告

> 文档元数据
> - 文件编号：14
> - 文档类型：research
> - 文件路径：docs/dev/14-research-file-operation-progress.md
> - 文档版本：v1.0.0
> - 最后更新：2026-06-26
> - 关联需求：文件复制、剪切和粘贴时显示总体圆形进度；点击展开每次粘贴的条形进度、当前文件、速度和取消按钮；跨文件系统剪切退化为复制后删除。

## 1. 问题与边界

- 问题描述：
  - 当前剪切粘贴只调用 `fs::rename`，跨文件系统时不会退化为复制后删除。
  - 当前复制/剪切粘贴通过一次性后台任务完成，没有可视化进度、详情展开或取消能力。
- 调研目的：
  - 明确现有复制/移动语义是否满足需求。
  - 设计符合用户指定 UI 的进度模型和最小实现边界。
- 包含：
  - 同文件系统剪切保留 `rename` 快路径。
  - 跨文件系统剪切在 `EXDEV` 时复制成功后删除源路径。
  - 复制和跨文件系统剪切提供总进度、每次粘贴进度、当前文件、速度和取消。
  - GUI 侧栏标题区域显示圆形总体进度入口，详情在其下方展开。
- 不包含：
  - 多选上下文菜单、专用桌面剪贴板 MIME target、暂停/恢复、失败重试队列。
  - 回收站能力。
- 非目标：
  - 不引入 DBus/GVFS/portal/桌面服务依赖。
  - 不改变现有新建、重命名、删除确认、打开方式逻辑。
- 禁止触碰范围：
  - 不对真实用户目录执行测试性复制、移动或删除。
  - 不修改系统配置、不调用 shell 拼接文件操作命令。

## 2. 当前证据

- 现有实现/现状：
  - `filesystem-core::paste_paths` 对 `PasteAction::Cut` 直接执行 `fs::rename`，未检查 `EXDEV`。
  - `filesystem-core::copy_entry` 递归复制目录，普通文件使用 `fs::copy`，没有分块进度。
  - GUI `ContextPaste` 优先使用内部剪贴板，再读取系统剪贴板文本；粘贴通过 `Task::perform` 一次性返回 `PasteFinished`。
  - GUI 已经使用 `Task::stream` 处理目录分批加载，可复用同类事件流模型。
- 已知约束：
  - 所有可能阻塞 UI 的文件系统操作必须在后台执行。
  - 复制/移动不覆盖已有路径，目标存在时自动唯一命名。
  - 删除无回收站，用户确认删除或取消文件操作后对已创建目标的清理需要限定在本次操作创建的目标路径内。
- 关键日志/数据/用户反馈：
  - 用户要求圆形进度入口放在侧栏分割线左侧的“文件”右侧，右对齐显示。
  - 圆形区域显示所有文件操作总进度，填充完中心显示对钩，延时三秒消失。
  - 点击圆形区域展开/隐藏详情；详情按每次粘贴显示独立条形进度、当前文件、复制/剪切速度和取消按钮。
- Bug 证据等级（按需）：E2。代码路径明确显示跨文件系统剪切未处理 `EXDEV`，已有同文件系统移动测试作为基线。
- 证据不足项：
  - 当前环境未准备两个真实不同文件系统做跨设备集成验证；可通过 core 纯逻辑单测验证 `EXDEV` 判定分支，真实跨设备保留人工验证风险。

## 3. 安全门禁摘要

| 项 | 结论 |
|----|------|
| 风险矩阵初判 | L4，涉及文件系统写操作、复制后删除、取消清理、后台进度流和 GUI 状态 |
| 命令权限 | C0/C1；验证只在项目和临时测试目录内执行 |
| 高风险开发门禁 | 是，文件系统副作用、错误路径、取消和后台异步任务 |
| 破坏性操作 | 运行时用户触发跨文件系统剪切会删除源路径；开发验证不触碰真实用户数据 |
| 用户已有修改 | 当前工作区干净 |
| 用户确认事项 | 用户已明确指定进度 UI 形态；取消按钮按常见文件管理器语义实现为停止该次操作并清理本次未完成目标 |

## 4. 候选方案

| 方案 | 核心思路 | 优点 | 风险/代价 | 适用条件 |
|------|----------|------|-----------|----------|
| A | 在 core 新增可回调进度的同步 API，GUI 用 `Task::stream` 桥接成事件流 | 无新依赖；保持 core 不依赖 iced；单元测试容易覆盖；符合既有后台模型 | 需要新增进度数据结构和取消 token；实现较当前一次性函数复杂 | 推荐用于当前需求 |
| B | 保留 core 一次性 API，只在 GUI 侧按文件数量模拟进度 | 改动少 | 不能显示大文件真实进度和速度；无法可靠取消；跨文件系统剪切仍需 core 改动 | 不满足需求 |
| C | 引入现成文件操作/进度库 | 可能减少部分代码 | 依赖与行为边界不可控；可能引入桌面服务假设；需要额外审计 | 当前少依赖目标下不采用 |

## 5. 推荐结论

- 推荐方案：方案 A。
- 取舍理由：
  - core 提供 `paste_paths_with_progress`，通过回调发出 `Started`、`Progress`、`Finished` 和 `Cancelled` 事件，并通过取消 token 做协作取消。
  - 同文件系统剪切先 `rename`，成功后视为该条目完成；仅在 `EXDEV` 时进入复制后删除路径。
  - 复制目录先预扫描统计总字节和文件数量，扫描阶段显示准备状态；复制普通文件时分块读写并累计字节。
  - GUI 只负责展示、聚合和发送取消，不直接执行文件 I/O。
- 需要进入 Plan 的关键约束：
  - 不覆盖已有目标，继续使用唯一命名。
  - 取消和失败清理只删除本次操作创建的目标根路径。
  - 粘贴成功后刷新当前目录；单个操作失败不影响其它已独立启动的粘贴操作状态。
  - 已完成操作保留 3 秒后从侧栏圆形进度入口中移除；有失败/取消项时保留详情，除非用户点 x 关闭。
- 需要用户确认的问题：
  - 无阻塞问题；用户已确认详情展开/隐藏和取消入口。
- 后续验证方向：
  - core 临时目录测试覆盖复制进度、取消、同文件系统 rename、跨设备错误判定分支。
  - GUI 编译和单测覆盖进度聚合/详情状态纯逻辑。
